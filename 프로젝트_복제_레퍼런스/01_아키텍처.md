# 아키텍처

## 전체 구조

앱은 세 층으로 나뉜다.

- Renderer: 사용자 입력, 이미지 목록, 생성 결과 편집, 버튼 상태 관리
- Preload: Renderer가 호출할 수 있는 안전한 API 노출
- Main: Electron 창, 파일 선택, AI 생성, Playwright 브라우저 자동화 담당

## 흐름

1. 사용자가 Renderer 화면에서 이미지와 기획값을 입력한다.
2. Renderer가 내부 상태를 workflow draft로 만든다.
3. 생성 요청 시 preload API를 통해 main process로 payload를 보낸다.
4. Main의 generation service가 Gemini API 작업을 생성하고 job ID를 반환한다.
5. 사용자가 결과 조회를 누르면 job 결과를 가져와 제목/본문/태그로 나눈다.
6. 최종 제출 시 draft model이 Naver payload를 만든다.
7. Main의 naver blog service가 글쓰기 페이지를 보장한다.
8. naver editor service가 iframe 내부 에디터에 블록을 순서대로 입력한다.

## 프로세스별 책임

Renderer 책임:

- DOM 이벤트 연결
- 현재 step, busy 상태, 이미지 목록, 기획값, 생성 결과 상태 보관
- 이미지 카드 렌더링
- 생성 결과 편집 필드 관리
- 제출 가능 여부 계산
- draft model 호출

Preload 책임:

- Renderer에 필요한 API만 노출
- IPC 채널명을 Renderer에서 숨김
- 드래그 앤 드롭 File 객체에서 실제 로컬 경로 추출
- status 이벤트 구독/해제 제공

Main 책임:

- 창 생성
- Playwright 브라우저 세션 생성
- 파일 선택 dialog
- 로컬 이미지 asset 생성
- AI 생성 service 생성
- 네이버 블로그 service 생성
- clipboard 쓰기

## 설계 포인트

- Renderer는 네이버 에디터 조작을 모른다.
- Main은 화면 입력 DOM 구조를 모른다.
- draft model이 두 영역 사이의 변환 책임을 가진다.
- Playwright 자동화는 `naverEditorService`에 몰아넣어 변경 영향 범위를 줄인다.
- AI 생성은 즉시 결과를 기다리지 않고 job으로 처리한다.
